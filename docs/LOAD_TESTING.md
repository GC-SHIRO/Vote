# 投票系统高并发场景压测指南

> **产品目标**：确保在营销活动推送、社媒集中引流等“洪峰”时刻，系统能够平稳承载同时 **1000 名用户**在几分钟内涌入页面并完成首屏渲染、选项查看与投票交互。任何明显的卡顿（超过1秒）或报错都会导致用户流失。

## 一、 为什么要做这次压测？

在线投票系统的核心挑战在于**读写瞬间的高密度**：
- **读扩散**：1000人同时打开页面，后端需要瞬时提供1000次选项列表。
- **写热点**：所有人的“投票”动作最后都落到数据库的局部几行数据上，产生排队和锁竞争。

我们期望通过模拟真实用户行为，观测当前架构在服务器 `8.145.62.223` 上的真实表现，提前暴露系统宕机临界点。

---

## 二、 压测工具 (k6) 及脚本说明

我们选用 **k6** 作为压测工具。它能够用极低的资源成本模拟庞大的用户群体。

### 1. 压测脚本位置
脚本已生成至：`backend/scripts/load-test.js`

### 2. 压测模型（流量起伏节奏）
脚本并不是一上来就砸1000个并发，而是模拟真实的传播曲线：
1. **预热期（30秒内增至200人）**：活动刚发出，用户陆续点击。
2. **爆发期（1分钟内飙升至1000人）**：大V转发，流量短峰爆发。
3. **高燃期（持续2分钟的1000人在线）**：大批用户在页面内疯狂点击和刷新。
4. **冷却期（30秒内降至0人）**：用户转化完毕，流量退潮。

## 三、 实操步骤指南

### Step 1: 在本地或测试机安装 k6
不需要在被压测的服务器(`8.145.62.223`)上安装，推荐在你的本地工作机上运行，以模拟外部真实网络请求。
- **Windows**: 在 PowerShell 运行 `winget install k6`
- **Mac**: 运行 `brew install k6`
- **Linux**: 参考 [k6 安装文档](https://k6.io/docs/get-started/installation/)

### Step 2: 确认目标服务器状态
正式压测前，请确保 `8.145.62.223` 服务器：
- 后端以生产模式启动（如使用 PM2 集群架构）。
- 已关闭不需要的调试日志（避免 I/O 拖垮性能）。
- Redis 和数据库已正常连接且开启服务。

### Step 3: 开始“流量轰炸”
在你的本地终端，进入项目后端目录执行脚本：
```bash
cd backend/scripts
k6 run load-test.js
```

## 四、 我们应该重点看什么？（观测与指标）

压测结束后，k6 会在终端生成一份报告。作为产品和研发侧，我们核心关注两项指标：

1. **`http_req_duration` (响应耗时)**
   - **指标含义**：用户点击后多久能看到结果。
   - **底线要求**：`p(95) < 500ms`。即 95% 的人必须在 0.5 秒内得到反馈。如果耗时达到 2~3 秒，会导致用户疯狂重试，进一步压垮服务器。
2. **`http_req_failed` (失败率)**
   - **指标含义**：多少人看到了“网络开小差”的错误提示。
   - **底线要求**：`< 1%`。如果错误率飙高，通常是因为数据库连接耗尽或者 Nginx/Node.js 被打满导致拒绝服务。

### 遇到瓶颈该如何应对？（常见策略）

如果压测未达标，无需过度深挖底层框架代码，从业务架构上调整见效最快：
- **遇到“读”瓶颈**（页面加载慢）：不要放任请求去查数据库，使用当前的 Redis 将投票选项缓存起来。
- **遇到“写”瓶颈**（投票报错500）：采用“接客与处理分离”模式。接收到用户投票先扔进 Redis 队列并立刻告诉用户“投票成功”，后台再慢慢排队处理入库（削峰填谷）。
